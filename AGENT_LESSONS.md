# 建 Agent 核心经验总结

> 基于「简历优化 Agent」项目实战提炼，为后续 Agent 项目（岗位管理、薪酬管理等）提供经验输入。

---

## 一、架构决策：多 Agent 拆分的边界怎么画

### 我们做对的事

按职责拆成 4 个 Agent（诊断/规划/优化/报告），每个 Agent 有独立的 system prompt、temperature、上下文策略。好处是可以**按角色调参**——诊断 Agent 用低温度(0.3)保证一致性，优化 Agent 用中温度(0.5)保留创造性。

### 踩过的坑

- OptimizeAgent 承担了太多职责（改简历 + 搜岗位 + 评真伪 + 认知引导），prompt 膨胀到 ~2500 行，导致 LLM 注意力稀释，指令遵循率下降
- Agent 之间缺乏共享状态，DiagnosisAgent 的判断结果要靠文本拼接传给 OptimizeAgent，信息会丢失

### 经验法则

> **一个 Agent 的 system prompt 超过 800 行就该拆了。** 拆分标准不是"功能多少"，而是"是否需要不同的思考模式"——分析型任务和生成型任务就该分开。

---

## 二、Prompt 工程：结构化 > 长篇大论

### 我们做对的事

- OptimizeAgent 有明确的用户意图分类机制（A/B/C/D 四类输入检测），让 Agent 先判断用户想干什么再行动
- 用 STAR 结构（背景-任务-行动-结果）作为简历改写的统一框架

### 踩过的坑

- 早期 prompt 写成"作文"，LLM 经常忽略中间段落的指令
- 反幻觉规则（"不要编造数据"）放在 prompt 末尾，效果远不如放在开头

### 经验法则

> **Prompt 结构应该是：角色定义 → 红线规则（不可违反）→ 输入格式 → 处理流程 → 输出格式。** 红线规则永远放最前面，因为 LLM 对开头和结尾的注意力最强，中间段最容易被忽略。

---

## 三、工具设计：Agent 的手和脚

### 我们做对的事

- 工具有明确的输入输出 schema（OpenAI function calling 格式），Agent 不需要猜参数
- 搜索工具有降级策略：Google API 失败 → LLM 生成合成结果
- 工具有上下文注入：tool executor 能访问对话记忆和原始评测结果

### 踩过的坑

- `re_evaluate_resume` 工具是同步阻塞的（调用收敛引擎要 10-20 秒），期间用户干等着
- LLM 降级搜索会产生虚假公司名和岗位——**降级不等于可以编造**
- 10 个工具全塞在一个 `ToolExecutor` 类里，945 行代码，改一个工具容易影响其他的

### 经验法则

> 1. **耗时 >3 秒的工具必须异步化**，给用户中间状态反馈（如"正在搜索中..."）
> 2. **降级策略要标注来源**——合成数据必须标记"AI 生成，仅供参考"，否则是在帮 Agent 撒谎
> 3. **工具按领域分组**：搜索类、简历操作类、评估类各一个 executor，单个文件不超过 300 行

---

## 四、记忆管理：最难也最关键的问题

### 我们做对的事

- 设计了 `ConversationMemory` 结构化记忆（已同意的修改、用户偏好、待解决问题），用 LLM 从对话中提取
- 历史压缩机制：超过 16 条消息时，保留最近 8 条原文，旧消息压缩成摘要
- 跨会话记忆：加载用户最近 5 次会话的摘要

### 踩过的坑

- 每轮对话都把完整简历（~3000 字）+ 8 个因素 + 全部历史发给 LLM，token 消耗失控
- 历史压缩由 LLM 做，压缩本身可能丢信息（"用户说不要改第三段"这种关键信息被压没了）
- 结构化记忆的提取也依赖 LLM，偶尔提取错误（把 Agent 的建议当成用户的同意）

### 经验法则

> 1. **分离"工作记忆"和"参考资料"**——简历全文是参考资料，只在需要时传入；用户的修改意图是工作记忆，每轮必带
> 2. **关键决策用确定性方式记录**，不要靠 LLM 事后提取。用户说"同意"时，代码层面直接写入 `agreed_modifications`，不要等 LLM 总结
> 3. **设计 token 预算**：给每个 Agent 设一个上下文 token 上限，超了就主动裁剪，而不是等着爆

---

## 五、状态管理：别用正则做状态机

### 我们做对的事

- 有明确的会话阶段（opening → optimizing → summary）
- TTL 过期自动清理会话

### 踩过的坑

- 阶段转换靠关键词匹配（`SUMMARY_KEYWORDS = ["总结", "结束", ...]`），用户随口说"总结一下刚才的修改"就会误触发结束流程
- 会话状态存内存，服务重启全丢
- 文件存储邀请码状态（JSON 文件），并发写入有竞态条件

### 经验法则

> 1. **状态转换要有确认机制**——我们后来加了二次确认（"你确定要结束吗？"），这个从一开始就该有
> 2. **持久化从第一天就要设计**，不要先用内存"临时方案"再说。Redis 或数据库，选一个，从 Day 1 用上
> 3. **别把业务状态存文件**——用数据库，或者至少用带锁的原子操作

## 六、LLM 调用策略：不是所有任务都需要最强模型

### 我们做对的事

- 双模型策略：推理模型（reasoner）用于精确的因素提取，聊天模型（chat）用于日常对话
- 不同 Agent 不同温度

### 踩过的坑

- 初期所有调用都用同一个模型同一个温度，评测结果波动大
- JSON 解析频繁失败——LLM 返回的 JSON 外面经常包着 markdown 代码块，需要多层清洗

### 经验法则

> 1. **按任务选模型**：结构化提取用推理模型 + `response_format: json_object`；对话生成用聊天模型 + 自然语言输出
> 2. **JSON 输出永远要有清洗层**——不要相信 LLM 会返回干净的 JSON，至少处理：markdown 代码块包裹、多余逗号、注释、截断
> 3. **给 LLM 的系统性偏差留修正空间**——我们发现 chat 模型对 PK（专业知识）评级偏保守，加了 +0.5 的修正。这种校准要基于数据，不要猜

---

## 七、错误处理：优雅降级 vs 默默失败

### 我们做对的事

- 评测失败时有 fallback 值，不会返回空结果
- 搜索失败时自动切换到 LLM 合成

### 踩过的坑

- Fallback 评测结果看起来和真实结果一样，用户完全不知道拿到的是假数据——**这是最严重的设计失误**
- 全局 `except Exception` 吞掉了所有错误细节，排查问题极其困难

### 经验法则

> 1. **降级必须告知用户**——返回 fallback 数据时，response 里必须有 `"degraded": true` 和降级原因
> 2. **错误分三级**：可重试的（网络超时）→ 自动重试；可降级的（搜索失败）→ 降级 + 标记；不可恢复的（API key 无效）→ 直接报错
> 3. **日志要结构化**——`logger.error(f"评测失败: {e}")` 远不如 `logger.error("assessment_failed", extra={"user_id": uid, "error_type": type(e).__name__, "input_length": len(text)})`

---

## 八、反幻觉：Agent 最容易翻车的地方

### 我们做对的事

- OptimizeAgent prompt 中有明确的反幻觉规则："不得编造用户简历中不存在的经历、数据、公司名"
- 设计了 `ReflectionChecker`，在后台用 LLM 校验改写结果是否捏造了信息

### 踩过的坑

- ReflectionChecker 是异步的，警告比改写结果晚 1-2 轮才出现，用户可能已经接受了错误内容
- Agent 在"帮用户润色"和"编造数据"之间的边界模糊——"帮你把销售额从100万改成200万"算润色还是造假？

### 经验法则

> 1. **反幻觉检查必须同步**——宁可慢 2 秒出结果，也不要先输出再纠正
> 2. **区分"润色"和"编造"的规则要写死在 prompt 里**：可以调整表述方式和结构，不可以修改数字、公司名、职位、时间线
> 3. **对于高风险工具（如搜索），降级生成的内容必须打标签**，前端用不同样式展示

## 九、新 Agent 项目的 Checklist

启动一个新 Agent 项目时，按这个清单过一遍：

### 架构设计阶段

- [ ] 明确 Agent 的核心任务边界（一句话能说清楚它干什么）
- [ ] 每个 Agent 的 prompt 控制在 800 行以内
- [ ] 定义显式状态枚举（不用关键词匹配做状态转换）
- [ ] 设计 token 预算（每轮上下文 token 上限）
- [ ] 选择持久化方案（数据库，不是内存/文件）

### Prompt 设计阶段

- [ ] 结构：角色 → 红线 → 输入 → 流程 → 输出
- [ ] 红线规则放在 prompt 最前面
- [ ] 定义用户意图分类（Agent 先判断意图再行动）
- [ ] 写明确的反幻觉规则（什么可以改，什么不能改）

### 工具设计阶段

- [ ] 每个工具有完整的 input/output JSON schema
- [ ] 耗时 >3s 的工具异步化
- [ ] 降级策略标注来源（AI 生成 vs 真实数据）
- [ ] 工具按领域分文件，单文件 <300 行

### 记忆设计阶段

- [ ] 分离工作记忆（每轮必带）和参考资料（按需加载）
- [ ] 关键决策代码级记录，不依赖 LLM 提取
- [ ] 设计历史压缩策略 + 保护关键信息不被压缩

### 错误处理阶段

- [ ] 降级返回标记 `degraded: true`
- [ ] 错误分级：可重试 / 可降级 / 不可恢复
- [ ] 结构化日志（不是 f-string 拼接）
- [ ] 反幻觉检查同步执行

## 十、简历 Agent vs 未来 Agent 的架构对照

| 维度 | 简历 Agent 的做法 | 建议改进（未来项目） |
|------|-------------------|----------------------|
| Agent 拆分 | 4 个 Agent，OptimizeAgent 过重 | 每个 Agent 职责单一，prompt < 800 行 |
| 工具注册 | 静态列表，一次性定义全部工具 | 动态注册，按 Agent 按需加载 |
| 记忆 | LLM 提取结构化记忆 | 关键决策代码级记录 + LLM 补充提取 |
| 持久化 | 内存为主 + 异步 Supabase | Day 1 就用数据库，内存只做缓存 |
| 上下文 | 全量传入每轮 | Token 预算制，按需裁剪 |
| 错误处理 | 静默降级（用户不知道拿到假数据） | 降级必标记，错误分级处理 |
| 状态机 | 关键词正则匹配 | 显式状态枚举 + 二次确认机制 |
| 反幻觉 | 异步检查，警告延迟 | 同步检查，宁慢勿错 |
| JSON 解析 | 多层清洗兜底 | 清洗层标准化，所有 Agent 复用 |
| 模型选择 | 双模型（推理+聊天） | 保持，按任务精细化选模型和温度 |
